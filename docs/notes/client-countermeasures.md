# SNS クライアントによるストレス低減対策

作成日: 2026-04-15  
参照: [sns-stress-research.md](sns-stress-research.md)

> このドキュメントは「sns-stress-research.md」で整理したストレス要因それぞれに対して、**SNS クライアントアプリとして実施できる対策**を検討したものである。SNS プラットフォーム側（投稿内容・サーバーアルゴリズム）は変えられないという前提のもと、**クライアント側の表示・操作・体験設計**で何ができるかに絞って考える。

---

## クライアントができることの範囲

対策を考える前に、クライアントアプリとプラットフォームの役割分担を整理しておく。

| 制御できる | 制御できない |
|---|---|
| 情報をどう表示するか | 他者が何を投稿するか |
| 何を表示しないか（フィルタ） | プラットフォームのアルゴリズム推薦 |
| いつ通知を届けるか | 相手の行動（既読・返信） |
| どんな操作を促すか | プラットフォームの利用規約・API 制限 |
| 使用量・感情状態の可視化 | インターネット全体の情報環境 |

---

## 1. 社会的比較・嫉妬への対策

**ストレスの根本:** 他者の「キュレーションされた輝き」が常に流れ込み、意図せず上方比較が起きる。

### 対策

**数値の非表示化（Metric Hiding）**

いいね数・リツイート数・フォロワー数は「誰が多いか」という比較を自動的に誘発する。これらの数値をデフォルトで非表示にするか、ユーザーが任意で隠せるオプションを設ける。Instagram が行ったいいね数非表示のテストでは、不安感の低下が報告されている。

- 投稿のエンゲージメント数（いいね・RT・引用数）の非表示オプション
- フォロワー数・フォロー数の非表示オプション
- 数値を見たいときだけ展開できる「折り畳み表示」

**時系列タイムラインの優先**

アルゴリズム推薦はエンゲージメントの高い（＝バズった・最も輝いた）投稿を上位に出す。クライアント側で取得した投稿を時系列に並べ直すことで、「平均的な投稿」も同等に表示される環境を作る。

- デフォルトをクロノロジカル順に設定
- アルゴリズム推薦タブ（「おすすめ」「トレンド」）をオフにするオプション
- 未フォローアカウントの投稿をタイムラインに挿入しない設定

**フォロー整理の支援**

比較を引き起こすアカウント（見るたびに劣等感を覚えるアカウント）を容易に遠ざけられる手段を提供する。

- 長押し・右クリックで即座にミュート・フォロー解除できる操作性
- 「しばらく見ない（30日ミュート）」などの一時的な距離置きオプション
- フォロー中アカウントの閲覧頻度や感情的反応の傾向を可視化するウェルネス機能（将来的）

## 2. 承認欲求・いいね依存への対策

**ストレスの根本:** 可変報酬スケジュールによって「いいね」の通知が依存を生む。自己価値が外部の数字に乗っ取られる。

### 対策

**通知のバッチ配信（Notification Batching）**

可変報酬の威力は「いつ報酬が来るかわからない」不規則性にある。通知をリアルタイムではなく、1 時間ごと・1 日 3 回など**まとめて届ける**ことで、その不規則性を壊し、依存形成を抑制できる。

- 通知の配信タイミングをユーザーが指定できる設定
- 「まとめて通知」モード（例: 朝・昼・夜の 3 回）
- 深夜・集中時間帯の通知を自動停止するスケジュール設定

**自分の投稿への反応数の非表示**

送り手として「自分の投稿がどれだけ反応を得たか」という数値は特にドーパミン依存を形成しやすい。

- 自分の投稿に付いたいいね数を表示しないオプション
- 「誰がいいねしたか」の一覧を見る前に一手間かける設計（ワンクリックで展開しない）

**「投稿後の反応確認」への摩擦**

投稿した直後に何度もリロードして反応を確認する行動は典型的な依存パターン。

- 投稿直後の通知を一定時間（15 分など）遅延させる設定
- 「投稿した内容は届いています。あとはゆっくり待ちましょう」といった安心メッセージ

## 3. FOMO への対策

**ストレスの根本:** 「見ていない間に何かあったかもしれない」という欠乏感が強迫的な確認行動を生む。

### 対策

**「すべて追わなくていい」設計の徹底**

FOMO の一因は「見ていない投稿が溜まっている」という視覚的・認知的プレッシャー。

- 未読カウント・バッジを表示しないオプション
- 「ここまで読んだ」マーカー（それより上は読まなくていいことを明示）
- タイムラインを開くたびに最新投稿から表示するのではなく、**前回終了した場所から再開する**オプション

**フィードの上限設定**

無限に続くフィードは FOMO を助長する。1 セッションあたりの表示件数に上限を設けることで「今日の分は読み終えた」という終了感を提供する。

- 1 日あたり・1 セッションあたりの表示件数の上限設定
- 上限に達したら「今日のフィードはここまでです」という終了画面

**ハイライト・ダイジェスト**

長時間離れていた後に「全部読まなければ」という焦りが生じやすい。大量の未読を圧縮して提示する機能が有効。ただし、この機能自体が「見逃した」感覚を強調するリスクもあるため、表現には配慮が必要。

- 長時間離席後の「最近の注目 3 件」程度のコンパクトなサマリー表示
- サマリーを見るかどうかはユーザーが選択（強制しない）

## 4. ドゥームスクロールへの対策

**ストレスの根本:** ネガティビティバイアス × 無限スクロール設計が強迫的なスクロールを生む。

### 対策

**無限スクロールの廃止**

無限スクロールは最もドゥームスクロールを助長するインターフェース。物理的な終端を設けることで自然な区切りを作る。

- 1 ページあたりの投稿数を固定し、次のページは明示的な操作（ボタン）で読み込む**ページネーション設計**
- 「一定数スクロールしたら一時停止」機能（スクロール量の物理的上限）

**ネガティブコンテンツのフィルタリング**

- ユーザー定義のキーワードミュート（ネガティブな話題を除外）
- センシティブ・暴力的・論争的なコンテンツを折り畳み表示（クリックしないと展開しない）
- 「今日はネガティブな話題を見たくない」モードの簡単な切替

**使用時間・スクロール量の可視化**

自分がどれだけスクロールしているか、どんな内容を見ているかを意識させることが行動変容に有効。

- セッション内のスクロール量・経過時間の表示
- 「X 分閲覧しました。一息つきますか？」という非強制的なブレークプロンプト
- 1 日の利用時間グラフ（週次レポートなど）

## 5. ハラスメント・炎上への対策

**ストレスの根本:** 攻撃的な返信・通知が 24 時間逃げ場なく届く。

### 対策

**通知の精密なフィルタリング**

- フォローしていないアカウントからの通知をデフォルトでブロックまたは別フォルダへ
- NGワード（攻撃的・差別的な語を含むリプライ）の自動フィルタリング
- 特定のアカウントからの接触を全遮断するブロック機能の強化（ブロックリストのインポート/エクスポート対応）

**炎上時の保護モード**

急激に通知が増えた場合（炎上兆候）に自動または手動で通知をバッファリングする。

- 「短時間に大量の通知が来ています。まとめて後で確認しますか？」という保護プロンプト
- 「通知を一時停止」ボタンへの素早いアクセス（常に 1 タップで届く位置に配置）
- 自分への言及（メンション・引用）を完全にオフにして**閲覧専用モード**へ移行する機能

**共有ブロックリストの活用**

コミュニティが共有する既知の荒らし・スパムアカウントのブロックリスト（Mastodon などのフェデレーション環境では一般的）をインポートできる仕組み。

## 6. 情報過多・SNS 疲れへの対策

**ストレスの根本:** 処理できる量を超えた情報・コミュニケーション・社会的期待が認知・感情を消耗させる。

### 対策

**リストとフィルターによる情報の絞り込み**

「全部見なければ」という強迫観念を解体する。

- フォロー中のアカウントをリストに分類し、場面ごとに「今日は仕事関係だけ見る」など切り替えられる
- キーワードフィルターで特定トピックだけを表示する絞り込みモード
- 「ハイライトのみ」モード（反応が多い投稿、または手動でスターを付けたアカウントのみ表示）

**コミュニケーション過多の緩和**

研究では情報過多よりコミュニケーション過多がより強く疲れを引き起こすと示されている。

- DM・リプライの優先度分類（重要・通常・後で）
- 「今は休憩中」ステータスの設定（メッセージが届くが通知はしない）
- グループチャットの通知をデフォルトでミュートし、確認したいときだけ開く設計

**「今日はここまで」機能**

- 主観的な疲労感を申告できるボタン（「今日はもう十分です」）→ その日のフィードを非表示
- 利用時間が設定上限に達したら穏やかに終了を促すメッセージ

## 7. フィルターバブル・エコーチェンバーへの対策

**ストレスの根本:** アルゴリズムが既存の信念を強化し続け、知らぬ間に世界認識が歪む。

### 対策

**アルゴリズム推薦の排除・透明化**

- デフォルトのタイムラインをアルゴリズムなしの時系列に（推薦フィードはオプトイン）
- 「なぜこの投稿が表示されているのか」の説明を表示
- 「おすすめユーザー」「トレンド」タブの完全非表示オプション

**複数プラットフォーム・複数アカウントの統合**

単一プラットフォームの閉じた世界に閉じこもらないよう、多様な情報源への接触を支援する。

- 複数の SNS アカウント（Mastodon・Bluesky・X など）を 1 つの画面で管理
- 同一トピックを複数プラットフォームで横断検索する機能

**意図的な多様性の提示（オプション）**

ユーザーが望む場合に限り、自分のフォロー圏外の視点を提示する。ただし強制せず完全にオプトインで。

## 8. パッシブ消費への対策

**ストレスの根本:** 一方的に眺めるだけの消費は疑似的な接続感を与えながら孤独感を増大させる。

### 対策

**アクティブな交流への誘導（低摩擦化）**

「眺めるだけ」より「誰かと話す」行動への心理的ハードルを下げる。

- 返信・DM 送信を 1 タップで始められる設計（返信欄の常時表示）
- 「この投稿、誰かに共有する？」というソフトな促し（強制しない）
- 特定のアカウントを「積極的に話しかけたい人」リストに追加し、その人の投稿への返信を後押し

**閲覧行動の可視化**

「今日何人と交流したか / 何件眺めただけだったか」を把握することで、パッシブ利用の自覚が生まれる。

- 閲覧のみ・いいね・返信・DM 送信の比率を週次で表示するウェルネスダッシュボード

**閲覧モードと交流モードの分離**

目的を意識させることで、「なんとなく眺める」から「誰かと話すために開く」への行動変容を促す。

- 起動時に「今日は何をしたいですか？（情報確認 / 誰かと話す / 特定の話題を見る）」を軽く聞く
- モードによって表示される画面を変える（交流モードでは DM・リプライを前面に出す）

## 9. 常時接続プレッシャーへの対策

**ストレスの根本:** 「いつでも返信できる状態であること」という暗黙の社会的期待が義務感と疲弊を生む。

### 対策

**既読表示・オンラインステータスの制御**

- 既読表示をオフにするオプション（相手に「読んだ」ことを知らせない）
- 自分のオンライン・最終ログイン時刻を非表示にするオプション
- 相手の「最終ログイン」表示を見ないようにするオプション（返信を待つ焦りを防ぐ）

**返信の緊急性を下げる UX ライティング**

UI の言語・表現が無意識のプレッシャーを生む。「たった今」「3 分前」という表示は「すぐ返せるはずだ」という期待を高める。

- 相対時刻（「3 分前」）を曖昧な表現（「今日」「先ほど」）に切り替えるオプション
- メッセージ受信時に「急がなくていい」を示唆するデフォルト文言（返信欄のプレースホルダーなど）

**不在モード・フォーカスモード**

- 時間帯を指定して通知を完全遮断する「フォーカスモード」
- 「今は休憩中です。後ほど返信します」という自動返信メッセージの設定（DM に対して）
- 曜日・時間帯のスケジュールで自動的にオン/オフが切り替わる設定

## 10. 横断的な設計思想

個々の対策の背景にある、アプリ全体を貫く設計哲学を整理する。

### ユーザー主権（User Sovereignty）

アルゴリズム・通知・エンゲージメント指標はすべて**プラットフォームの利益**のために設計されている。このアプリは**ユーザーの意図と幸福**のために設計する。ユーザーが「何をどのくらい見るか」を自分で決められることを最優先にする。

### 摩擦の再設計（Intentional Friction）

中毒的な行動（無限スクロール・通知の確認・いいね数のチェック）には意図的に**摩擦を加え**、健全な行動（返信・DM・閲覧の終了）には**摩擦を下げる**。良い習慣は簡単に、悪い習慣はちょっと面倒に。

### 透明性と自己認識（Transparency）

ストレスの多くは「気づかないうちに蓄積する」。どれだけ使ったか、何を見ているか、どんな感情状態か——これらを可視化してユーザーの自己認識を促す。

### 「離れること」の肯定（Normalizing Disconnection）

「SNS を閉じること」「返信しないこと」「見逃すこと」を罪悪感なくできるよう、UI・UX ライティングの全体で一貫してサポートする。アプリを閉じるときに「また後で」「ゆっくり休んでください」といったメッセージを表示するなど、**オフラインを歓迎する姿勢**を設計に組み込む。

### ウェルネスの計測（Wellbeing Metrics）

エンゲージメント時間ではなく、ユーザーが「使った後に気分が良かったか」を重視する。利用後の気分を簡単に記録できる機能を設け、使用パターンと気分の相関を可視化する（長期的にはアプリの改善にも活用）。

## まとめ：対策一覧表

| ストレス要因 | 主な対策 | 実装難度 |
|---|---|---|
| 社会的比較・嫉妬 | エンゲージメント数の非表示、時系列タイムライン、簡易ミュート | 低〜中 |
| 承認欲求・いいね依存 | 通知のバッチ配信、自分の反応数の非表示、投稿後の遅延通知 | 中 |
| FOMO | 未読カウント非表示、ページ上限設定、「ここまで読んだ」マーカー | 低〜中 |
| ドゥームスクロール | ページネーション設計、スクロール量の上限、キーワードフィルタ | 中 |
| ハラスメント・炎上 | フォロー外通知のフィルタ、炎上保護モード、閲覧専用モード | 中〜高 |
| 情報過多・SNS 疲れ | リストによる絞り込み、DM 優先度分類、利用時間上限設定 | 中 |
| フィルターバブル | 時系列タイムライン強制、複数プラットフォーム統合 | 中〜高 |
| パッシブ消費 | 返信への低摩擦化、ウェルネスダッシュボード、モード切替 | 中〜高 |
| パラソーシャル関係 | インフルエンサー投稿の分離・限定表示オプション | 中 |
| 常時接続プレッシャー | 既読表示・ステータス非表示、フォーカスモード、時刻表示の曖昧化 | 低〜中 |

> **実装難度の基準:**  
> 低 = API から取得したデータの表示を変えるだけで実現可能  
> 中 = クライアント側でのデータ処理・状態管理・設定 UI が必要  
> 高 = 複数プラットフォームへの対応や機械学習的な処理が必要

## 参考：先行する取り組み

既存のクライアント・プラットフォームでこれらの一部はすでに試みられている。

- **Instagram のいいね数非表示テスト**（2019〜）— 比較からの解放を目的とした実験
- **Apple の Screen Time / Android の Digital Wellbeing** — 使用時間の可視化と上限設定
- **Ivory / Tweetbot 等の Mastodon クライアント** — アルゴリズム推薦なし・広告なし・時系列タイムライン
- **BeReal** — リアルタイム・加工なし投稿によるキュレーション文化への対抗（ただし成否は検証中）
- **Firefox Focus** — プライバシーと「見えないトラッキング」への対抗（設計思想の参考として）
