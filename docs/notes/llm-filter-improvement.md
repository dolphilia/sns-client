# ローカル LLM フィルター 速度・精度改善メモ

作成日: 2026-04-19  
参照: [filtering-technologies.md](filtering-technologies.md)

> prototype-03 に実装した Ollama + qwen3.5:9b によるストレス検出フィルターについて、速度・精度それぞれの観点から改善策を整理する。M1 Pro / 16GB 環境を前提とする。

---

## 現状のアーキテクチャ

```
投稿テキスト
    │
    ▼
【Layer 0】キーワード照合 (keywordFilter.ts)     ~0ms
    │
    ▼
【Layer 1】mDeBERTa ゼロショット分類 (mlFilter)  ~500ms（WASM）
    │
    ▼
【Layer 2】Ollama LLM / qwen3.5:9b              ~3〜5秒/件（逐次）
```

**実測値（2026-04-18 時点）**

- モデル: `qwen3.5:9b`（Q4_K_M、6.6GB）
- バックエンド: Metal（MLX は 32GB 以上が必要なため不使用）
- think:false 適用後の平均レイテンシ: **約 3〜5 秒/件**
- バッチ 8 件を逐次処理した場合の合計: **約 30〜40 秒**

**判明した問題点**

| 問題 | 原因 | 対処済み |
|------|------|---------|
| 全件スコア 0.00 | qwen3.5 は思考モデルのため `content` が空、`num_predict` 上限内で思考だけが終わった | `think: false` を追加 |
| スコアが整数（8 など）で返る | モデルが 0〜10 スケールで出力した | プロンプト強化 ＋ `/10` 正規化 |

---

## 速度改善

### S-1. 並列リクエスト 【優先度：高】

**概要**  
現状は投稿を 1 件ずつ順番に Ollama へ送信している。Promise.all で複数件を同時送信することで、バッチ処理時間が大幅に短縮できる。

**M1 Pro 16GB での現実的な同時実行数**: 2〜3 件（メモリ・GPU スレッドの制約）

```ts
// 現状（逐次）
for (const text of posts) {
  results.push(await ollamaFilterService.analyzeDetailed(text));
}

// 改善案（並列、同時 2 件）
const chunks = chunkArray(posts, 2);
for (const chunk of chunks) {
  const batch = await Promise.all(chunk.map(t => ollamaFilterService.analyzeDetailed(t)));
  results.push(...batch);
}
```

**期待効果**: バッチ 8 件の合計時間を約 1/2 に短縮。

---

### S-2. 段階フィルタリング（LLM 呼び出しを絞る） 【優先度：高】

**概要**  
既存の Layer 0・Layer 1 が確実に判定できた投稿（確信度の高い高スコア・低スコア）は LLM を呼ばない。LLM は判定が難しい中間域の投稿だけに使う。

```
Layer 0  キーワード → スコア 1.0   → LLM スキップ（フィルタ確定）
Layer 0  キーワード → スコア 0.0   → LLM スキップ（通過確定）
Layer 1  mDeBERTa  → スコア ≥ 0.8 → LLM スキップ（フィルタ確定）
Layer 1  mDeBERTa  → スコア ≤ 0.2 → LLM スキップ（通過確定）
Layer 1  mDeBERTa  → 0.2 〜 0.8   → LLM で精密判定
```

**期待効果**: 実運用では投稿の多数が明確にクリーンなため、LLM の呼び出し回数を全体の 10〜20% 程度に抑えられる可能性がある。

---

### S-3. 小型モデルとの速度・精度比較 【優先度：高】

**概要**  
qwen3.5:9b より小さいモデルへの切り替えで大幅な速度向上が見込める。精度との実測トレードオフを比較して最適モデルを選定する。

| モデル | VRAM 消費 | 推定速度 | 日本語力 |
|--------|----------|---------|---------|
| qwen3.5:9b（現状） | ~6.6GB | ~4秒/件 | ★★★★★ |
| qwen3.5:4b | ~3GB | ~1.5秒/件 | ★★★★ |
| qwen3.5:2b | ~1.5GB | ~0.8秒/件 | ★★★ |
| gemma4:4b | ~3GB | ~1.5秒/件 | ★★★★ |

**実験方法**: 研究ページ（`/research/ollama`）の `MODEL_NAME` 定数を切り替えてバッチテストを実行し、スコアの妥当性とレイテンシを比較する。

---

### S-4. `num_predict` の削減 【優先度：低】

**概要**  
現状 `num_predict: 300` だが、実際の JSON レスポンスは 80〜120 トークン程度。150 に下げることで推論打ち切りが早くなる。

```ts
options: {
  temperature: 0.1,
  num_predict: 150,  // 300 → 150
}
```

**期待効果**: 件あたり数百ms の短縮。効果は小さいが実装コストがゼロ。

---

### S-5. Ollama サーバーの環境変数チューニング 【優先度：低】

**概要**  
brew services の環境変数で Metal バックエンドの動作を最適化できる。

```bash
OLLAMA_FLASH_ATTENTION=1   # Flash Attention で推論高速化
OLLAMA_KV_CACHE_TYPE=q8_0  # KV キャッシュを量子化してメモリ節約
OLLAMA_NUM_PARALLEL=2      # 最大同時処理数
```

設定方法: `/etc/launchd.conf` または `~/Library/LaunchAgents/` の plist に追記。

## 精度改善

### A-1. Few-shot プロンプト 【優先度：高】

**概要**  
現状のプロンプトはゼロショット（例示なし）。具体例を数件加えるだけでスコアのスケール感が安定し、「全部 0.8」「全部 0.3」のような偏りが解消される。

**追加例（プロンプト末尾に挿入）**:

```
【出力例】
投稿: 今日は天気がよくて気持ちいい
→ {"stress_score": 0.02, "category": "問題なし", "reason": "日常的なポジティブな投稿。", "confidence": "high"}

投稿: 〇〇人は出て行け
→ {"stress_score": 0.95, "category": "ヘイトスピーチ", "reason": "特定の民族・属性への差別的な排除表現。", "confidence": "high"}

投稿: 政治家は全員嘘つきだ
→ {"stress_score": 0.50, "category": "論争的主張", "reason": "政治的な一般化だが暴力的表現はない。", "confidence": "medium"}
```

**期待効果**: スコアの分布が自然になり、閾値設定がしやすくなる。実装コストはほぼゼロ。

---

### A-2. 多次元スコアリング 【優先度：中】

**概要**  
1 つの `stress_score` では「攻撃性」「不安煽り」「論争性」を同列に扱ってしまう。次元を分けることで、ユーザーが自分の感受性に合わせた細かい設定が可能になる。

**出力フォーマット案**:

```json
{
  "toxicity":    0.9,
  "anxiety":     0.2,
  "controversy": 0.5,
  "self_harm":   0.0
}
```

**ユーザー設定への応用**:  
設定画面で「攻撃的な投稿は除外するが政治議論は見たい」という個人差に対応できる。prototype-03 の `FilterSettings` インターフェースの拡張が必要。

---

### A-3. Confidence-based routing 【優先度：中】

**概要**  
モデルが `"confidence": "low"` を返したケースに対して、処理を分岐させる。

```
confidence: high   → 判定をそのまま採用
confidence: medium → キーワード・mDeBERTa との多数決
confidence: low    → think: true で再解析（精度優先・低速）
                     または「判定不能」として非表示
```

**実装上の注意**: think を有効にすると現状 15〜20 秒かかるため、実用上は「判定不能として非表示」がより現実的。

---

### A-4. System / User ロール分離 【優先度：低】

**概要**  
現状は指示とテキストを 1 つの user メッセージにまとめている。system ロールに指示を分けることで、モデルの文脈理解が安定しやすくなる。

```ts
messages: [
  {
    role: "system",
    content: "あなたはSNSストレス検出AIです。...",
  },
  {
    role: "user",
    content: `以下の投稿を分析してください：\n${text}`,
  },
]
```

---

### A-5. プロンプト言語の実験 【優先度：低】

**概要**  
多言語モデルへの指示に英語を使うと出力が安定するケースがある。「英語 system プロンプト ＋ 日本語投稿テキスト」の組み合わせが精度向上につながる可能性がある。研究ページで A/B 比較を実施する。

---

### A-6. 文脈考慮（将来的な拡張） 【優先度：低】

**概要**  
現状は投稿を 1 件単体で判定しているが、リプライチェーンの文脈によって同じテキストのストレス度が変化する。

```
単体:「それはひどい」         → stress_score: 0.20
文脈（加害的な流れの中):「それはひどい」 → stress_score: 0.70
```

実装には AT Protocol の `reply` フィールドを辿って親投稿を取得し、プロンプトに含める処理が必要。

---

## 改善ロードマップ（推奨順）

```
Phase 1 — 即時実施（実装コスト：低）
  ├─ A-1. Few-shot プロンプト追加
  ├─ S-4. num_predict を 150 に削減
  └─ S-3. 小型モデル（qwen3.5:4b, gemma4:4b）との比較実験

Phase 2 — 短期（実装コスト：中）
  ├─ S-1. 並列リクエスト（同時 2 件）
  ├─ S-2. 段階フィルタリング（LLM 呼び出しを絞る）
  └─ A-3. Confidence-based routing

Phase 3 — 中期（実装コスト：高）
  ├─ A-2. 多次元スコアリングと設定 UI の拡張
  ├─ S-5. Ollama サーバーチューニング
  └─ A-4. System / User ロール分離・プロンプト言語比較

Phase 4 — 将来（実装コスト：非常に高）
  └─ A-6. リプライ文脈を考慮した判定
```

## 最終的なアーキテクチャ構想

```
投稿テキスト
    │
    ▼
【Layer 0】キーワード照合              ~0ms
    │ 確実な判定以外
    ▼
【Layer 1】mDeBERTa（WASM）            ~500ms
    │ スコア 0.2〜0.8 のグレーゾーンのみ
    ▼
【Layer 2】Ollama LLM（並列・2件同時）  ~2〜3秒
    │ confidence: low のケースのみ
    ▼
【Layer 3】判定不能として折り畳み表示
```

このパイプラインにより、投稿の大半（~80%）は Layer 0〜1 で処理され、LLM の呼び出しはごく一部に限定される。
