# Bluesky 以外の SNS クライアント候補調査

作成日: 2026-06-29

## 目的

現行アプリは Bluesky / AT Protocol 向けに作っているが、同じ「ストレスを減らすデスクトップ SNS クライアント」というコンセプトを他 SNS に適用する場合の候補を整理する。

評価軸は以下とする。

- 作りやすさ: API の公開度、認証、タイムライン取得、投稿・反応操作、既存 TypeScript 実装との相性
- ユーザー規模: 公開されているアクティブユーザー数または推定値
- コンセプト適合: エンゲージメント数非表示、自動更新抑制、通知圧低減、ローカル保存などを実装しやすいか
- 継続リスク: API 課金、利用規約変更、プラットフォーム都合での締め出しリスク

## 結論

最有力は **Mastodon / ActivityPub**。次点で **Misskey**。日本語圏の関心という意味では **mixi2** も気になるが、現時点では公開された外部クライアント向け API が確認できず、非公式 API 解析やスクレイピング前提の実装は避けるべき。規模を重視するなら **Threads** は候補になるが、現時点ではフル機能の代替クライアントとしては API 制約が大きい。**Reddit** はコミュニティ閲覧クライアントとしては面白いが、第三者クライアントの事業リスクが大きい。**X** はユーザー規模は大きいものの、API コスト・制限・方針変更リスクが高く、このプロジェクトの初期候補からは外すのが妥当。

## 1. Mastodon / ActivityPub

### 評価

Mastodon は最も現実的な候補。理由は、REST API、OAuth、タイムライン、投稿、通知、ブックマーク、ミュート、ブロックなどがクライアント実装向けに公開されており、現在の Bluesky クライアント構造と対応付けやすいため。

現行アプリの抽象に置き換える場合も、以下の対応がしやすい。

| 現行 Bluesky 機能 | Mastodon での対応 |
|---|---|
| ホームフィード | Home timeline |
| 通知一覧 | Notifications |
| 投稿詳細・スレッド | Status context |
| いいね | Favourite |
| ブックマーク | Mastodon 側の Bookmark またはローカル保存 |
| リポスト抑制 | Boost 表示の抑制 |
| ミュート / ブロック | API で対応可能 |

### 良い点

- サードパーティクライアント文化があり、API がクライアント向けに整備されている。
- アルゴリズム推薦・広告圧が比較的弱く、「ストレスを減らす」方向と相性がよい。
- インスタンスを選べるため、ユーザーがコミュニティ境界を意識しやすい。
- ブックマーク、ミュート、フィルター、コンテンツ警告など、ストレス低減に使いやすい機能が多い。

### 悪い点

- ユーザー規模は Threads / Reddit / X よりかなり小さい。
- インスタンスごとにバージョン、文字数、許可機能、モデレーション、レート制限が違う。
- 初回ログイン時に「どのサーバーか」を入力させる必要があり、Bluesky よりオンボーディングが重い。

### 実装方針

最小 MVP は「Mastodon 互換クライアント」として始めるのがよい。ActivityPub 全体を直接実装するより、まず Mastodon REST API に寄せる方が現実的。

## 2. Misskey

### 評価

Misskey は日本語圏との相性がよく、UI 文化も Bluesky / Mastodon と近い。公開 API と Streaming API があり、クライアント開発は可能。ただし Mastodon API とは別物なので、ActivityPub 対応と一括りにせず、Misskey 専用アダプタとして考える必要がある。

### 良い点

- 日本語圏ユーザーとの相性がよい。
- リアクション、カスタム絵文字、アンテナ、クリップなど独自機能が豊富。
- API は公開されており、OAuth / MiAuth / Streaming API が用意されている。
- Misskey 公式は PWA 中心で、専用デスクトップクライアントには余地がある。

### 悪い点

- Mastodon より API と UI モデルの独自性が強い。
- リアクションや MFM をどこまで実装するかで複雑度が上がる。
- インスタンス差が大きい。
- 全体のアクティブ規模は大手中央集権 SNS より小さい。

### 実装方針

Mastodon と共通化するより、`SocialAdapter` のような境界を置き、Misskey は別アダプタとして実装する方が安全。最初はホームタイムライン、投稿詳細、リアクション表示、ローカルブックマークまでに絞る。

## 3. mixi2

### 評価

mixi2 は日本語圏向け候補としては気になるが、現時点ではクライアントアプリの対象として優先しない方がよい。公式サイトでは「今を共有でき、すぐ集える」というコンセプト、ポスト、フォロー、通知、コミュニティ、イベント、リアクション、リポスト、引用、ブックマーク、投票など、SNS クライアントとして扱いたくなる機能が確認できる。一方で、外部開発者向けの公開 API ドキュメントは確認できなかった。

利用規約では、MIXI ID による会員登録、公式アプリの使用許諾、リバースエンジニアリング禁止、不正な連続アクセス禁止、不正使用を目的としたツールやプログラムの開発・配布・使用禁止、コンテンツ等の無断転載・複製・蓄積・転送等の禁止が定められている。そのため、非公式 API 解析やスクレイピングでデスクトップクライアントを作るのは、技術的にできてもプロジェクト方針としては採用しない方がよい。

### 良い点

- 日本語圏ユーザー向けの親和性が高い。
- ポスト、フォロー、通知、コミュニティ、イベント、リアクションなど、SNS クライアントとして扱いたい機能は豊富。
- 公式 Web 版も存在するため、機能モデルの観察はしやすい。
- 招待・コミュニティ・イベント中心の設計は、単なる無限フィードより「集まり」を重視する別方向のアプリ設計に応用できる。

### 悪い点

- 公開 API、OAuth クライアント登録、開発者向けドキュメントが確認できない。
- 公式 MAU や DAU が見つからず、アクティブ規模を評価しづらい。
- 非公式通信解析やスクレイピング前提の実装は、規約・運用・保守のリスクが高い。
- 新規登録は公式アプリ導線が中心で、外部デスクトップクライアントのオンボーディングを設計しづらい。

### 実装方針

現時点では対象外。作るとしても、公式 API が公開されるまでは「mixi2 風の日本語圏向け SNS 体験研究」や「コミュニティ・イベント導線の UI 研究」に留める。実サービス接続クライアントとしては、Mastodon / Misskey の方がかなり現実的。

## 4. Threads

### 評価

ユーザー規模では最も魅力的な候補。2026 年 6 月に 5 億 MAU を超えたと報じられており、Bluesky / Mastodon より桁違いに大きい。ただし、API はブランド・投稿管理・分析・返信管理寄りで、一般ユーザーが日常的に使う「完全な代替クライアント」を作れるかは慎重に見る必要がある。

### 良い点

- ユーザー規模が非常に大きい。
- テキスト中心 SNS として、現行アプリのカード UI・フィード UIを転用しやすい。
- ActivityPub 連携が進んでおり、将来 Fediverse 側から一部接続できる可能性がある。

### 悪い点

- Meta プラットフォーム依存が強い。
- API がサードパーティ閲覧クライアント向けに十分開いているとは言いにくい。
- アルゴリズム推薦・Instagram / Facebook 連携・広告など、ストレス低減コンセプトと衝突しやすい要素がある。

### 実装方針

Threads 単体クライアントではなく、「投稿補助」「返信確認」「自分の投稿管理」寄りのツールなら現実味がある。日常閲覧クライアントとしては API の追加開放待ち。

## 5. Reddit

### 評価

Reddit は巨大なコミュニティ閲覧対象で、SNS 疲れ軽減クライアントとしては面白い。特に「購読 subreddit だけを静かに読む」「スコアを隠す」「コメント深掘りを制限する」「ローカル保存する」といった体験はこのプロジェクトに合う。

ただし、2023 年の API 課金変更で Apollo など主要第三者クライアントが終了しており、商用・大規模クライアントとしての継続リスクは高い。

### 良い点

- アクティブユーザー規模が大きい。
- API ドキュメント上は listing、投稿、コメント、投票、保存、subreddit などクライアントに必要な要素が揃っている。
- 「コミュニティ単位で読む」設計はストレス低減と相性がよい。

### 悪い点

- API 課金・商用制限リスクが大きい。
- 投稿スコア、ランキング、炎上コメントなど、比較・報酬・対立を促す設計が強い。
- 日本語圏での一般 SNS クライアント需要は Bluesky / Mastodon より読みづらい。

### 実装方針

作るなら「Reddit クライアント」ではなく、「購読コミュニティを静かに読むリーダー」として切り出す方がよい。

## 6. Tumblr

### 評価

Tumblr はブログ / リブログ文化が中心で、短文 SNS クライアントというより創作・日記・画像投稿の閲覧体験に近い。API は存在するが、現行アプリの Bluesky 風タイムラインをそのまま移植する対象としては優先度が低い。

### 良い点

- ブログ、画像、創作、長文との相性がよい。
- リブログ文化は「反応数を隠して読む」設計と組み合わせる余地がある。

### 悪い点

- ActivityPub / WordPress 移行計画が保留されており、将来の基盤が読みづらい。
- SNS というよりブログ寄りで、現行アプリのホームフィード設計とはズレる。
- 現在のアクティブユーザー規模が読みづらい。

### 実装方針

優先しない。将来やるなら「静かな Tumblr リーダー」または創作閲覧アプリとして別設計にする。

## 7. Nostr

### 評価

Nostr は技術実験としては面白い。中央 API の許可に依存せず、WebSocket relay からイベントを読む構造なので、クライアント開発の自由度は高い。一方で、一般ユーザー向け SNS クライアントとしては鍵管理、スパム、リレー選択、モデレーション、検索の説明コストが高い。

### 良い点

- オープンプロトコルで、特定企業の API 方針に縛られにくい。
- クライアント開発の自由度が高い。
- ローカル重視、ユーザー主導の設計と相性がよい。

### 悪い点

- 一般ユーザー向けの導入が難しい。
- スパム対策とモデレーションをクライアント側でも考える必要がある。
- アクティブユーザー規模が読みづらく、コミュニティも偏りがある。

### 実装方針

現行アプリの次候補ではなく、研究用プロトタイプ向き。

## 8. X

### 評価

ユーザー規模だけなら候補に入るが、現実的には非推奨。API の課金、レート、方針変更、リンク投稿課金などの不確実性が大きく、個人・小規模開発で安定した第三者クライアントを維持する前提に向かない。

### 良い点

- 利用者規模とリアルタイム性は大きい。
- 旧 Twitter クライアントの UI パターンを応用しやすい。

### 悪い点

- API コストと制限が重い。
- サードパーティクライアントへの信頼性が低い。
- エンゲージメント競争、炎上、アルゴリズム推薦が強く、コンセプトと衝突しやすい。

### 実装方針

対象外でよい。どうしても扱うなら、公式 API を前提にした限定的な投稿管理・分析ツールに留める。

## 推奨ロードマップ

1. 現行 Bluesky 実装を完成させる。
2. その後、抽象化のために `TimelineItem` / `Actor` / `PostAction` / `SocialAdapter` の境界を薄く作る。
3. 2 番目の対応先として Mastodon を選ぶ。
4. Mastodon 対応が安定したら、Misskey を別アダプタとして検証する。
5. mixi2 は公式 API が公開されるまで、実サービス接続先としては保留する。
6. Threads / Reddit は API 制約と利用規約を再調査してから別プロダクトとして判断する。

## 実装優先順位

### 含める

- Mastodon ホームタイムライン
- 投稿詳細と返信ツリー
- 投稿、削除、いいね、ブーストの最小操作
- ブックマーク、ミュート、ブロック
- エンゲージメント数の非表示
- 自動更新しないフィード

### 後回し

- Mastodon 複数アカウント
- Misskey 独自機能
- Fediverse 横断検索
- mixi2 連携
- Threads 連携
- Reddit 連携

### 除外

- X フルクライアント
- mixi2 非公式 API 解析・スクレイピング前提のクライアント
- ActivityPub プロトコル直接実装からの開始
- 全 SNS 統合タイムラインの初期実装
