# Prototype-03 設計メモ：エディションモデル × 完了の美学

## 概要

prototype-01・02 で「表示の設計」「見るコンテンツの設計」に取り組んだ。  
prototype-03 では **「使い方そのものの構造を変える」** レイヤーに踏み込む。

具体的には、SNS の常時接続型モデルを根本から問い直す。

| 現在のSNSモデル | エディションモデル |
|---|---|
| フィードが常時ライブ更新される | フィードはスナップショットとして固定される |
| 終わりがなく、いつでも新しい投稿が来る | 「号」という単位で届き、読み終わりがある |
| 止まるための理由がない | 完了した瞬間が体験として設計されている |

---

## 1. 解決したいストレス要因

このプロトタイプが主に対処する要因（研究文書より）：

| 要因 | メカニズム | 本プロトタイプでの対処 |
|---|---|---|
| **FOMO** | 「見逃しているかも」という強迫的確認 | 号という単位で「全部読んだ」を実現可能にする |
| **ドゥームスクロール** | 無限の先があるから止まれない | 号の底があるから自然に止まれる |
| **承認欲求依存** | 投稿後の反応を何度も確認しに戻る | 号が固定されているため「また開く」コストが上がる |
| **常時接続プレッシャー** | アプリが常に最新状態を求めてくる | ユーザーが「取得する」まで更新されない |
| **パッシブ消費** | 目的なく惰性で読み続ける | 「読み始める」という能動的な操作が起点になる |

## 2. エディションモデルの設計

### 2-1. 「号」とは何か

**号（Edition）** とは、ユーザーが「読み始める」操作をした瞬間に作成されるフィードのスナップショット。

- デフォルト **30件** の投稿で構成される（設定でカスタマイズ可能）
- 一度取得したら固定。バックグラウンドで自動更新されない
- 号を読み終えるか、ユーザーが「閉じる」まで有効
- 次の号は、ユーザーが明示的に「新しい号を取得する」操作をして初めて作られる

```
[取得操作] → 号が作成（スナップショット）
     ↓
[読書体験]  号の中を読み進める
     ↓
[完了状態]  全部読む、または「閉じる」を選ぶ
     ↓
[待機状態]  次の号を取得するまでアプリは静か
```

### 2-2. 号の取得フロー

**起動時の状態分岐：**

```
アプリ起動
  ├─ 未読の号がある          → 号の続きを表示（前回の読み位置から）
  ├─ 号を読み終えた後         → 完了画面（静かな状態）
  └─ 号がない（初回 or 完了後） → 待機画面 → 「読み始める」ボタン
```

**「読み始める」操作：**
- Bluesky API (`getTimeline`) を呼び、最新N件を取得してローカルに保存
- 取得後は API を呼ばない（号が有効な間はオフライン状態でも読める）
- バックグラウンドで「新しい投稿が届いている」ことだけを検知可能（オプション）

### 2-3. 読書体験の UX

**読み進め方：**
- 投稿は縦方向にリスト表示（prototype-01・02 と同じ構造）
- ページネーションではなく、号の中は連続スクロールで読める（号そのものが有限）
- スクロールの底に到達すると完了状態へ遷移

**プログレスの表示（数字を使わない）：**
- 画面の端に細いプログレスバーのみ（幅だけで進行を表現）
- 「残り○件」「○%完了」は一切表示しない
- 視覚的な奥行き：号の終盤に近づくにつれて投稿のコントラストが微妙に落ち着く

```
                              [━━━━━━━━━━░░░░░] ← プログレスバー（幅のみ）
  ┌─────────────────────────────────────────┐
  │  投稿 A                                 │
  ├─────────────────────────────────────────┤
  │  投稿 B                                 │
  ├─────────────────────────────────────────┤
  │  ...                                    │
  └─────────────────────────────────────────┘
```

## 3. 完了の美学

### 3-1. 完了状態の定義

3種類の完了がある：

| 種類 | 発生条件 | 意味 |
|---|---|---|
| **自然完了** | 号の最後まで読んだ | 届いた分を全部読んだ |
| **任意完了** | 「今日はここまで」ボタンを押した | 途中でも十分と感じた |
| **保留** | アプリを閉じた（読み途中） | 続きはまた後で |

### 3-2. 完了画面のデザイン

**原則：**
- 数字を一切出さない（「○件読んだ」「今日○分使った」など全て不要）
- 余白が主役。テキストは最小限
- アニメーションは静かに（フェードイン、動きは穏やかに）
- 「次を読め」という誘導をしない

**自然完了の画面：**

```
─────────────────────────────────────────────────
 
 
      読み終えました。
 
 
  [次のフィードを取得する]        [閉じる]
 
 
─────────────────────────────────────────────────
```

- 背景は白または非常に淡い色
- テキストは中央配置、フォントは細め
- ボタン2つは主張しない（小さく、枠線のみなど）
- 「次のフィードを取得する」は右配置、「閉じる」は左配置（閉じる方を主動線にしない）

**任意完了の画面：**

```
─────────────────────────────────────────────────
 
 
      今日はここまで。
 
 
  [続きを読む]                    [閉じる]
 
 
─────────────────────────────────────────────────
```

- 自然完了と同じトーン
- 「続きを読む」がある（途中なので）が、配置は同様にクローズ動線を優先しない

### 3-3. 完了へのアクセス

**「今日はここまで」ボタン：**
- タイムライン画面の隅に常時存在（目立たない場所に）
- ヘッダーやフッターに溶け込むような控えめな配置
- 押すと任意完了画面へ

```
  ┌──────────────────────────────────────────────┐
  │ [←] タイムライン                    […|終]   │ ← 「終」が「今日はここまで」
  ├──────────────────────────────────────────────┤
  │  投稿 A                                      │
  │  投稿 B                                      │
  │  ...                                         │
  └──────────────────────────────────────────────┘
```

### 3-4. 完了後の待機状態

完了後にアプリを閉じず、そのままにした場合の画面：

```
─────────────────────────────────────────────────

          [時間帯に応じた背景色]

      （余白のみ。何も促さない。）

  [新しいフィードを取得する]

─────────────────────────────────────────────────
```

- アプリが「もっと見ろ」と言わない
- ユーザーが能動的に次を取りに行くまで、この静かな画面が続く

## 4. 補助要素：環境との同期

エディションモデル・完了の美学と自然に組み合わさる要素として、軽く実装を検討する。

### 4-1. 時間帯による UI の変容

| 時間帯 | UI の質感 | 背景色の方向性 |
|---|---|---|
| 朝（5〜10時） | 明るく、清涼感 | 淡い白〜水色系 |
| 昼（10〜17時） | ニュートラル | 標準（白系） |
| 夕方（17〜21時） | 落ち着いた暖色 | 淡いアンバー系 |
| 夜（21〜0時） | 穏やかに暗い | ライトグレー〜薄いネイビー |
| 深夜（0〜5時） | 非常に静か | 暗めのトーン |

**実装方針：**
- `new Date().getHours()` で時間帯を取得
- CSS カスタムプロパティ（`--background`, `--surface` 等）を時間帯ごとに切り替え
- Tailwind の設定で時間帯クラスを管理
- アプリ起動時と、号を取得したタイミングで時間帯を再評価

**重要：** 時刻そのものは画面に表示しない。環境同期は「数字なしに時間帯を感じさせる」ための仕組み。

## 5. プロトタイプの状態管理設計

### 5-1. 新規ストア：editionStore

```typescript
type Edition = {
  id: string                 // UUID
  posts: PostView[]          // 号のコンテンツ（固定）
  createdAt: Date            // 取得日時（内部管理用。UIには出さない）
  readUntilIndex: number     // どこまで読んだか
  status: 'reading' | 'completed' | 'paused'
}

type EditionStore = {
  currentEdition: Edition | null
  fetchEdition: () => Promise<void>
  markRead: (index: number) => void
  completeEdition: () => void
}
```

### 5-2. 既存ストアへの影響

| ストア | 変更内容 |
|---|---|
| `authStore` | 変更なし |
| `settingsStore` | エディション設定を追加（号のサイズ, 環境同期ON/OFF など） |
| `filterStore` | 変更なし（号のコンテンツにフィルタリングを適用） |

### 5-3. フィルタリングとの統合

prototype-02 のフィルタリング機能は号の中のコンテンツに対して引き続き適用する。

```
号取得（API）
  ↓
Layer 1 フィルタ（キーワード）
  ↓
Layer 2 フィルタ（ML推論、Web Worker）
  ↓
フィルタ済みの号を表示
```

## 7. 検証したいこと（判断軸）

prototype-03 を実際に使って確認すること：

**エディションモデルについて：**
- 「読み始める」という操作が儀式として心地よいか、それとも面倒か
- 号が固定されていることで「新しい情報を逃している」感が出るか
- 号の適切なサイズ感はどのくらいか（30件か、50件か）
- 「また後で読もう」という保留の体験は自然か

**完了の美学について：**
- 完了画面で「満足した」「終わった」という感覚が得られるか
- 「今日はここまで」ボタンを実際に使いたくなるか
- 完了後の待機画面は静かさを感じるか、空虚さを感じるか

**環境同期について：**
- 時間帯による色変化が「時刻を伝えられている」感を与えるか（それは避けたい）
- 朝・夜の質感の違いが使い方のリズムに影響するか

**インタラクションについて：**
- いいね（数字なし）は承認欲求への影響が少ないか。押すことへの意味が変わるか
- ブックマークは「後で読み返す」という行為として機能するか、ほとんど使われないか
- リポスト・返信がなくても「操作できることが少なすぎる」と感じるか

**全体として：**
- prototype-01・02 と比べて「もっと見たい」という衝動が減っているか
- 「読み終わった」体験が一日の中で繰り返し訪れることで、利用パターンが変わるか
- 操作の選択肢が少ないことが「すっきり感」か「物足りなさ」かどちらに感じるか

## 9. 決定済み事項

### 通知
- アプリの通知機能は **デフォルト OFF**
- 実装するとしてもアプリのアップデートなど「本当に重要なこと」のみ
- 「新しい投稿が届いています」等の号・投稿関連通知は実装しない
- Bluesky の通知タブは引き続き存在するが、バッジ・プッシュ通知は不要

### 号のサイズ
- デフォルト **30件**
- 設定画面からカスタマイズ可能（例：10・20・30・50件の選択肢）
- settingsStore の `editionSize` として管理

### 自分の投稿の扱い
- 自分の投稿は受け取る号（タイムライン）には **含めない**
- 自分の投稿は別の場所に蓄積する：**マイポスト号**（後述）

## 10. マイポスト号の設計（自分の投稿の扱い）

### コンセプト

自分が投稿したものを「受け取る」フィードに混ぜない。代わりに、自分の投稿は **短い日記を書き溜めていくような場所** に蓄積する。

```
タイムライン号（受け取る場所）
  → 他の人の投稿が届く。読んで完了する。

マイポスト号（発信した場所）
  → 自分が書いたものが時系列で積み重なる。
  → 振り返る・見直すための場所。
  → SNS のフィードではなく、ポートフォリオや日記に近い質感。
```

### なぜ分けるか

- タイムラインに自分の投稿が混在すると「自分の投稿への反応が気になる」循環が生まれる
- 自分の投稿を「受け取る場所」ではなく「発信した記録」として扱うことで、承認欲求依存を構造的に切り離す
- 発信行為が「会話への参加」ではなく「日記を書く」という意味づけになる

### マイポスト号の UI

- サイドバーに「自分の投稿」ページとして独立
- タイムライン号と同じく有限のリスト表示（無限スクロールしない）
- いいね数・リポスト数は非表示（デフォルト、prototype-01 の設定に準拠）
- 時系列の並び（古い順 or 新しい順を選べる）

## 11. インタラクションモデル

### ユーザーが行える操作（確定）

prototype-03 で投稿に対して行えるアクションは **いいね** と **ブックマーク** の2つのみ。

| アクション | Bluesky との同期 | 相手への通知 | 実装方法 |
|---|---|---|---|
| **いいね** | あり（`app.bsky.feed.like` を呼ぶ） | あり（相手の通知に届く） | prototype-01・02 と同様 |
| **ブックマーク** | なし（ローカル保存のみ） | なし | `bookmarkStore` にURIを保存 |

### 実装しないアクション

| アクション | 理由 |
|---|---|
| **返信** | 直接コミュニケーションよりも間接コミュニケーションが主流。号の読書体験を分断する |
| **リポスト** | 「増幅」という行為がエンゲージメント・承認欲求ループの起点になる |
| **引用投稿** | 公開演説的な性質を持ち、炎上・pile-on の主要な起点になり得る |
| **投稿へのメモ** | 有意義な機能だが、このプロトタイプでは保留 |

### いいね について

いいね は相手に届く社会的シグナルであり、承認欲求ループを完全には断ち切れない。しかし以下の理由で実験的に実装する：

- prototype-01 からすでに実装済み（エンゲージメント数は非表示）
- 「押した / 押していない」という行為そのものは残す
- 自分のいいね数・相手のいいね数は引き続き非表示（デフォルト）
- 数字が見えないいいねが承認欲求に与える影響を体験として検証する

### ブックマーク について

ブックマークは純粋に個人的な行為。相手への影響がゼロ。

- 実装：`bookmarkStore`（Zustand + localStorage）に投稿の URI を保存
- Bluesky の API は呼ばない（ローカルのみ）
- 「保存した投稿」一覧ページを別途設ける
- 号の読書中に「後で読み返したい」投稿を静かに保存できる

```typescript
type BookmarkStore = {
  bookmarks: string[]              // 保存した投稿の URI リスト
  addBookmark: (uri: string) => void
  removeBookmark: (uri: string) => void
  isBookmarked: (uri: string) => boolean
}
```

### 受信側のリポスト表示

フォロー中の人がリポストした投稿が号に混入することへの対処：

- デフォルト設定：**フォロー中の人の原文のみ表示**（リポストを号に含めない）
- 設定でリポストを含めることも可能
- `settingsStore` の `editionIncludeReposts: boolean` で管理
