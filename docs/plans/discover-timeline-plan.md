## 目的

自分が興味を持ちそうな投稿をしている、まだフォローしていないユーザーを見つけるための **「発見」タイムライン** を追加する。

ただし、当面は AI / LLM による興味判定や重い推薦処理は使わない。ユーザーが明示的に選んだ Bluesky カスタムフィードを関心源とし、アプリ側では **未フォロー投稿の抽出・ミュートキーワード除外・軽い重複抑制** に絞る。

---

## 採用方針

### 採用する考え

| 項目 | 方針 |
|---|---|
| 機能名 | `発見` |
| 取得元 | ユーザーが保存した発見用フィード URI のうち、選択中の 1 件 |
| 入力形式 | `at://...` と `https://bsky.app/profile/.../feed/...` の両方を受け付ける |
| 推薦方法 | AI なし。フィード由来の投稿から未フォロー作者を抽出する |
| 表示対象 | 投稿主が自分のフォロー外である投稿 |
| 除外対象 | フォロー済み作者、自分自身、ミュートキーワードに一致する投稿 |
| 並び順 | フィード API の順序を基本にする |
| 投稿操作 | 既存の `PostCard` と `FollowButton` を流用する |
| 表示理由 | 控えめに `登録フィードから` と表示する余地を残す |

### 採用しない考え

| 項目 | 理由 |
|---|---|
| AI / LLM による興味判定 | 現方針では処理負荷と環境依存を避ける |
| いいね履歴からの類似推定 | 初期実装としては仕様が膨らむ |
| フォロー中ユーザーの交流圏分析 | 取得量と判断ロジックが増えるため後回し |
| 全件取得してローカルランキング | 重くなりやすく、発見画面の初期目的には過剰 |
| エンゲージメント数によるランキング | 比較圧を強めるため使わない |

## MVP スコープ

### 作るもの

1. 左サイドバーに `発見` メニューを追加する。
2. `/discover` ルートを追加する。
3. 設定画面に `発見に使うカスタムフィード` を追加する。
4. `agent.app.bsky.feed.getFeed({ feed, limit, cursor })` で投稿を取得する。
5. 投稿主がフォロー済みの場合は表示しない。
6. 投稿主が自分自身の場合は表示しない。
7. ミュートキーワードに一致する投稿は表示しない。
8. 同一作者の投稿が連続して画面を占有しないように、1 ページ内で軽く間引く。
9. 投稿カードにはフォロー / フォロー解除ボタンを表示する。
10. ページ下部到達時に過去投稿を自動で追加取得する。
11. フォールバックとして `もっと読む` ボタンを残す。

### 作らないもの

1. AI 判定。
2. 投稿本文の意味解析。
3. いいね履歴ベースの推薦。
4. フォローグラフの分析。
5. 複数フィード横断の高度なランキング。
6. 発見結果の永続キャッシュ。
7. 通知やバッジによる発見の促し。

## 設定設計

既存のホームフィード設定 `feedSettings` とは分けて、発見専用の設定を追加する。

```ts
interface DiscoverSettings {
  feedUris: string[];
  selectedFeedUri: string;
}
```

複数の URI を保存し、その中から 1 件だけを選択して `/discover` の取得元にする。保存値はユーザーが入力した文字列を保ち、取得直前に正規化する。

### デフォルト

| 項目 | 値 |
|---|---|
| `discoverSettings.feedUris` | `[]` |
| `discoverSettings.selectedFeedUri` | `""` |

### 設定画面

`設定 > フィード` 内に、ホームフィードとは別枠で以下を追加する。

- ラベル: `発見に使うカスタムフィード`
- 説明: `複数保存できます。選択中の 1 件から、まだフォローしていない投稿主の投稿を表示します。bsky.app のフィード URL もそのまま使えます。`
- placeholder: `https://bsky.app/profile/.../feed/... または at://...`

未登録または未選択の場合、`/discover` では設定を促す空状態を表示する。選択中の URI を削除した場合は、残っている先頭 URI を自動で選択する。
保存済みフィードは `app.bsky.feed.getFeedGenerator` で取得できた `displayName` を URI と併記する。名称取得に失敗した場合でも、URI は表示し続ける。

設定値は保存時には入力値を保ち、取得直前に `normalizeFeedUri` で正規化する。

```text
https://bsky.app/profile/did:plc:xxxx/feed/yyyy
  ↓
at://did:plc:xxxx/app.bsky.feed.generator/yyyy
```

## 取得・フィルタ設計

### データ取得

`useInfiniteQuery` を使い、Bluesky の cursor をそのまま扱う。

```text
/discover を開く
  ↓
discoverSettings.selectedFeedUri を読む
  ↓
getFeed(feed, limit, cursor)
  ↓
未フォロー・ミュートキーワード・自分自身を除外
  ↓
PostCard で表示
  ↓
下端到達で次ページ取得
```

### 除外条件

| 条件 | 判定 |
|---|---|
| フォロー済み作者 | `post.author.viewer?.following` がある |
| 自分自身 | `post.author.did === session.did` |
| ミュートキーワード | 投稿本文にキーワードが含まれる |
| 重複投稿 | 同じ `post.uri` が既に表示済み |
| 同一作者の偏り | 1 ページ内で同じ作者を最大 2 件程度に抑える |

### ミュートキーワード

既存の `settingsStore.mutedKeywords` を使う。大文字小文字は区別せず、単純な部分一致で判定する。

## UI 設計

### サイドバー

`ホーム` の下、または `自分の投稿` の上に `発見` を追加する。

候補:

```text
ホーム
発見
自分の投稿
いいね
繋がり
...
```

アイコンは Lucide の `Compass` を使う。

### 発見画面

ヘッダー:

- タイトル: `発見`
- 説明: `登録フィードから、まだフォローしていない投稿主を見つけます`
- 右上: `更新`

本文:

- 未設定時: 設定画面への導線を表示する
- 読み込み中: `読み込み中...`
- エラー時: `発見フィードの取得に失敗しました`
- 結果なし: `表示できる投稿がありません`
- 下端: `もっと読む` / `すべて表示しました`

投稿カード:

- 既存の `PostCard` を使う。
- `headerAction` に `FollowButton` を差し込む。
- 必要ならカード上部に小さく `登録フィードから` を追加できるが、MVP では省略してよい。

## 実装手順

1. `settingsStore` に `discoverSettings` と `setDiscoverSettings` を追加する。
2. persist の `merge` で古い設定と安全に統合する。
3. `settings.tsx` に発見用フィード URI の追加・削除・選択 UI を追加する。
4. bsky.app のフィード URL を AT URI に変換する `normalizeFeedUri` を追加する。
5. `Sidebar.tsx` に `/discover` を追加する。
6. `src/routes/discover.tsx` を追加する。
7. `/discover` で `useInfiniteQuery` を使って `getFeed` を呼ぶ。
8. 未フォロー、自分自身、ミュートキーワード、重複、同一作者偏りを除外する。
9. `PostCard` + `FollowButton` で投稿を表示する。
10. `IntersectionObserver` で下端到達時の自動追加取得を実装する。
11. `npm run routes:gen` で TanStack Router のルート型を再生成する。
12. `npm run build` で検証する。

## 判断メモ

この機能は「おすすめ」ではなく「発見」として扱う。アプリが強く推薦するのではなく、ユーザーが選んだ関心源から、まだフォローしていない投稿主を見つけやすくするための画面である。

そのため、初期実装ではランキング精度よりも、軽さ、説明しやすさ、設定の明示性を優先する。
