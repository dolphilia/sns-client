`prototypes/prototype-05/` で検証していた Electron ベースの実装を、正式なデスクトップアプリ本体として `apps/desktop/` に昇格する。

今後の実装・検証・ビルド対象は `apps/desktop/` とし、`prototypes/` は過去の実験を残す場所として扱う。大きな再設計は行わず、まずはディレクトリ、開発コマンド、ドキュメント、アプリ名を現行アプリ前提に整理する。

## 目的

このデスクトップアプリでは、これまでの実験で大きく通常の SNS から離れていた部分を整理し、**「ストレスを減らすが、普通の SNS として自然に使える Bluesky デスクトップクライアント」** に軌道修正する。

検証したいことは **3 つ**に絞る。

1. **Electron ベースのデスクトップ基盤への移行**  
   Tauri ではなく Electron を採用し、React + Vite の既存フロントエンドを活かしたデスクトップアプリ構成を確立する。

2. **AI / 号モデルなしの通常フィード体験**  
   prototype-03 / 04 の「号」「読み終わり」「AI 吟味」を外し、一般的な SNS に近い無限フィードを実装する。ただし、新着投稿の自動差し込みは行わず、過去投稿はスクロールで遡れるようにする。

3. **ストレス低減要素の最小セット確認**  
   エンゲージメント数非表示、通知圧の低減、交流操作の最小化、相手へ通知されないローカル操作を、通常の SNS 体験の中で違和感なく成立させる。

## 基本方針

| 方針 | 現行アプリでの扱い |
|---|---|
| 比較を減らす | 継続。いいね数・リポスト数・返信数はデフォルトで非表示。 |
| 終わりを作る | 見送り。号や完了画面ではなく、一般的な無限フィードへ寄せる。 |
| 自動更新しない | 継続。新着投稿を勝手にタイムラインへ差し込まない。 |
| 過去投稿の取得 | スクロールまたは明示操作で過去投稿を追加取得できる。 |
| 通知圧を下げる | 継続。通知バッジ・未読カウント・リアルタイム圧を抑える。 |
| 交流を最小化する | 継続。返信・リポスト・引用は前面に出しすぎない。 |
| ローカル操作を重視する | 継続。ブックマークはローカル保存とし、相手に通知されないことを明示する。 |
| 使い終わりを肯定する | 専用完了体験は見送り。通常 SNS の操作感を優先する。 |
| AI 判定に依存しない | 継続。投稿表示の可否を AI / LLM に依存させない。 |

| 機能 | 理由 |

| Tauri 基盤 | Electron へ移行するため。 |
| Next.js 化 | デスクトップ SNS クライアント本体には現時点で不要。 |
| AI / LLM 投稿吟味 | 処理負荷と環境依存が大きいため当面見送り。 |
| mDeBERTa / Transformers.js フィルター | AI 判定に依存しない方針へ戻すため。 |
| 号モデル | 通常の SNS から離れすぎるため当面見送り。 |
| 号サイズ設定 | 号モデル廃止に伴い不要。 |
| 完了画面 / 読み終わり体験 | 通常の無限フィードへ寄せるため当面見送り。 |
| 時間帯に応じた背景色 | 現方針では不要。 |
| プッシュ通知 / 通知バッジ | 通知圧を下げるため。 |
| リポスト / 引用投稿の強い導線 | 増幅行為や炎上導線を前面化しないため。 |
| OAuth | プロトタイプ段階では App Password で継続。 |

## フィード設計

### 基本動作

- 初回表示時に最新のホームフィードを取得する。
- 過去投稿はスクロール下端到達、または「さらに読み込む」で取得する。
- 新着投稿は自動で画面上部へ挿入しない。
- 新着確認を入れる場合は、ユーザーが押す控えめな「新しい投稿を確認」操作にする。
- 未読数や強いバッジ表現は使わない。

### ページング

TanStack Query の `useInfiniteQuery` を使い、Bluesky API の cursor をそのまま扱う。

```text
初回取得
  ↓
最新投稿を表示
  ↓
下方向スクロール
  ↓
cursor で過去投稿を追加取得
```

### 自動更新しない理由

新着投稿の自動挿入は、読んでいる位置をずらし、FOMO と常時接続感を強める。現行アプリでは、フィードは一般的な無限フィードに戻すが、更新タイミングはユーザーに委ねる。

## インタラクション設計

### 投稿カードの操作

| 操作 | Bluesky 同期 | 相手への通知 | UI 方針 |
|---|---|---|---|
| いいね | あり | あり | 使えるが、数字はデフォルトで隠す。 |
| ブックマーク | なし | なし | 「ローカル保存」「相手に通知されません」を明示する。 |
| 返信 | あり | あり | 詳細画面中心。ホームカードでは前面に出しすぎない。 |
| リポスト | あり | あり | 当面は非表示または詳細画面で検討。 |
| 引用投稿 | あり | あり | 当面は非表示。 |

### ローカル操作の表現

ブックマークなど、相手に通知されない操作はユーザーが安心して使えるようにする。

- ボタン tooltip: `ローカルに保存します。相手には通知されません。`
- ブックマーク一覧の説明: `この一覧はこの端末内に保存されています。`
- 設定画面に「ローカル操作」セクションを置く。

## 設定設計

設定ページのヘッダーは右カラムいっぱいに表示し、設定フォーム本体は広いウィンドウで横に広がりすぎないよう `max-w-5xl` 程度で中央寄せする。

### 残す設定

| 設定 | デフォルト | 説明 |
|---|---|---|
| エンゲージメント数を非表示 | true | いいね数・リポスト数・返信数を隠す。 |
| リポストをフィードから除外する | true | フォロー中の人の原文中心にする。 |
| 画像付きの投稿のみ表示 | false | ホームと発見で画像を含む投稿だけを表示する。 |
| NSFW 投稿を除外する | true | ホームと発見で NSFW 系ラベルのある投稿を表示しない。 |
| 投稿カードにミュートボタンを表示 | true | 各投稿の右下に、確認 dialog 付きのユーザーミュート操作を表示する。 |
| フォロー解除後の再フォローを防止 | true | フォロー解除したユーザーをローカルに記録し、未フォロー状態のフォローボタンを無効にする。 |
| フォロー解除時にミュートする | false | フォロー解除と同時に Bluesky のミュートも実行する。 |
| フィードソース | following | ホームフィードを基本にする。 |
| ホーム用カスタムフィード | 空 | 複数保存し、選択中の 1 件をホームのカスタム取得元にする。bsky.app のフィード URL も受け付け、設定欄では取得できたフィード名を表示する。 |
| フィードを幅いっぱいに表示 | true | オンの場合は投稿一覧部分を左サイドバーを除いた右カラム全体に表示する。ヘッダーは常に右カラム全体に表示する。 |
| フィードの幅 | 720 | 幅いっぱい表示がオフのときの投稿一覧部分の最大幅。 |
| 投稿をグリッド表示 | false | 投稿カードを複数列で表示する。 |
| グリッド投稿サイズ | 320 | グリッド表示時の投稿カード最小幅。 |
| 投稿カード表示項目 | すべて表示 | アバター、ユーザー名、ユーザーID、投稿日時、本文、画像、アクション領域を個別に表示 / 非表示できる。 |
| 画像を正方形にトリミング表示 | false | 投稿画像をカード幅に合わせ、中央で正方形に切り抜いて表示する。 |
| カスタムキーワード | 空 | AI ではない明示的なミュート補助として検討。 |

### 廃止する設定

| 設定 | 理由 |
|---|---|
| 号サイズ | 号モデル廃止。 |
| Ambient Sync | 時間帯背景を廃止。 |
| LLM モデル選択 | AI 吟味を見送り。 |
| 吟味の強さ | AI 吟味を見送り。 |

## 状態管理設計

### TanStack Query

| Query | 用途 |
|---|---|
| `timeline` | ホームフィードのページング取得。 |
| `notifications` | 通知一覧の取得。 |
| `profile` | プロフィール取得。 |
| `post-thread` | 投稿詳細・スレッド取得。 |
| `my-posts` | 自分の投稿一覧。 |
| `bookmarks` | 保存 URI から投稿を再取得。 |

### Zustand

| Store | 用途 |
|---|---|
| `authStore` | ログイン / ログアウト、暗号化されたセッション復元。 |
| `settingsStore` | ストレス低減設定、フィード設定。 |
| `bookmarkStore` | ローカルブックマーク。 |
| `unfollowedActorStore` | フォロー解除済みユーザーのローカル記録。 |

Electron 実行時は、`settingsStore` / `bookmarkStore` / `unfollowedActorStore` を `localStorage` ではなく、Electron の `app.getPath("userData")` 配下の `local-store/` に JSON ファイルとして保存する。ブラウザ開発実行時は localStorage を使う。

認証セッションを扱う `authStore` は通常の Zustand persist から外し、Electron 実行時は `safeStorage` で暗号化した上で `app.getPath("userData")` 配下の `secure-store/bsky-session.json` に保存する。既存の `localStorage` 内 `bsky-auth.session` は初回復元時に secure-store へ移行し、以後は `bsky-auth` から削除する。ブラウザ開発実行時は Electron の `safeStorage` を使えないため、開発用の localStorage フォールバックを使う。

Linux では Electron の `safeStorage.getSelectedStorageBackend()` が `basic_text` になる環境があり得るため、配布前には保存状態の警告表示または自動ログイン保存の無効化を検討する。

### 削除・非採用

| Store | 理由 |
|---|---|
| `screeningStore` | AI 吟味廃止。 |
| `editionStore` | 号モデル廃止。 |
| `filterStore` の ML 系 | AI / ML 判定に依存しない方針。 |

## 今後の製品化に向けた判断軸

- 通常の SNS として自然に使えるか
- エンゲージメント数非表示は不便ではなく、比較圧の低下として感じられるか
- 自動更新しないことは落ち着きにつながるか、それとも新着確認の不便さになるか
- 過去投稿を遡るフィード体験は十分に快適か
- 返信・リポスト・引用を前面に出さない設計は、すっきり感か、機能不足か
- ローカルブックマークの「相手に通知されない」明示は安心感につながるか
- Electron 化による開発体験・起動・ビルド・配布の見通しはよいか
