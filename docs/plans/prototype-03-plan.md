## 目的

このプロトタイプで検証したいことは **3 つ**に絞る。

1. **エディションモデルの体験検証**  
   フィードを「常時ライブ」から「スナップショット単位の号」へ切り替えることで、ドゥームスクロール・FOMO・パッシブ消費が実際に減るかを体感する。

2. **完了の美学の体験検証**  
   号を読み終えたときに「満足した・終わった」という感覚が生まれるか、完了状態のデザインが習慣的なスクロールを自然に断ち切れるかを確かめる。

3. **インタラクション最小化の影響確認**  
   返信・リポスト・引用投稿をなくし、いいね・ブックマークのみに絞った操作体系が「使いにくい」か「すっきりしている」か、どちらの印象を与えるかを体感する。

## エディションモデルの設計

### 号とは何か

**号（Edition）** とは、「読み始める」操作をした瞬間に作成されるフィードのスナップショット。

- `getTimeline` API を呼び、最新 N 件（デフォルト 30 件）を取得してローカルに保存
- 一度取得したら固定。バックグラウンドで自動更新されない
- 号が有効な間はオフライン状態でも読める
- 次の号は、ユーザーが明示的に「新しいフィードを取得する」を押して初めて作られる

### 起動時の状態分岐

```
アプリ起動
  ├─ 号あり（reading）  → 前回の読み位置から再開
  ├─ 号あり（completed）→ 完了後の待機画面
  └─ 号なし            → 待機画面 → 「読み始める」で号を取得
```

### 号の読書フロー

```
[読み始める] → 号を取得（スナップショット）
     ↓
[タイムライン] 号の中を連続スクロールで読む
     ├─ 「今日はここまで」ボタン → [任意完了画面]
     └─ スクロール底に到達      → [自然完了画面]
          ↓
[完了後の待機画面]
     ↓
[新しいフィードを取得する] → 次の号へ
```

### リポストのフィルタリング

フォロー中の人がリポストした投稿が号に混入することへの対処：

- デフォルト：**フォロー中の人の原文のみ**（リポストを号に含めない）
- 設定でリポストを含めることも可能
- `settingsStore` の `editionIncludeReposts: boolean` で管理

## 完了の美学

### 3 種類の完了状態

| 状態 | 発生条件 | 画面コピー |
|---|---|---|
| **自然完了** | 号の最後まで読んだ | 「読み終えました。」 |
| **任意完了** | 「今日はここまで」ボタンを押した | 「今日はここまで。」 |
| **保留** | アプリを閉じた（読み途中） | 次回起動時に続きから再開 |

### 完了画面のデザイン原則

- 数字を一切出さない（読んだ件数・経過時間など）
- 余白が主役。テキストは最小限
- アニメーションは静かに（フェードイン）
- 「次を読め」という誘導をしない
- 2 つのボタンは主張しない（小さく、枠線のみ）

```
─────────────────────────────────────────────────

      読み終えました。


  [閉じる]                [新しいフィードを取得する]

─────────────────────────────────────────────────
```

## 状態管理設計

### 新規ストア

**`src/stores/editionStore.ts`**

```typescript
type Edition = {
  id: string
  posts: PostView[]
  createdAt: Date           // 内部管理用。UIには表示しない
  readUntilIndex: number
  status: 'reading' | 'completed' | 'paused'
}

type EditionStore = {
  currentEdition: Edition | null
  fetchEdition: () => Promise<void>
  markRead: (index: number) => void
  completeEdition: () => void
}
```

## 補助機能：環境との同期

時刻は画面に表示しない。アプリの質感が自然に時間帯を反映する。

| 時間帯 | 質感 | 背景色の方向 |
|---|---|---|
| 朝（5〜10時） | 明るく清涼 | 淡い白〜水色系 |
| 昼（10〜17時） | ニュートラル | 標準（白系） |
| 夕方（17〜21時） | 落ち着いた暖色 | 淡いアンバー系 |
| 夜（21〜0時） | 穏やかに暗い | ライトグレー〜薄いネイビー |
| 深夜（0〜5時） | 非常に静か | 暗めのトーン |

**実装方針：**
- `new Date().getHours()` で時間帯を取得し、`<html>` に `data-time-of-day` 属性を付与
- CSS カスタムプロパティで背景色・テキスト色を切り替え
- 号を取得するタイミングで再評価
- `ambientSyncEnabled: false` のときは標準（昼）のトーンで固定

## フィルタリングとの統合

prototype-02 のフィルタリングをそのまま引き継ぐ。号取得後にフィルタを適用する。

```
号取得（getTimeline API）
  ↓
リポストを除外（editionIncludeReposts: false のとき）
  ↓
Layer 1 フィルタ（キーワード・正規表現）
  ↓
Layer 2 フィルタ（ML 推論、Web Worker）
  ↓
フィルタ済みの号を editionStore に保存
  ↓
EditionReader が表示
```

## 完成の定義（このプロトタイプが「終わった」と言える条件）

- 「読み始める」→ 号の読書 → 完了画面 の一連のフローが動作する
- 号が固定されている（取得操作なしに自動更新されない）
- 完了画面が静かに表示され、「終わった」感覚が得られる
- PostCard から返信・リポスト・引用ボタンが消えている
- ブックマークができ、一覧で確認できる
- マイポスト号で自分の投稿を振り返れる
- 1〜2 日実際に使ってみて、利用パターンの変化を感じられる

## 次のプロトタイプに向けた判断軸

**エディションモデルについて：**
- 「読み始める」操作は儀式として心地よいか、それとも面倒か
- 号が固定されていることで「新しい情報を逃している」感が出るか
- 30 件というサイズ感は「読み切れる量」か、多すぎるか少なすぎるか
- 「保留（途中でアプリを閉じる）」の体験は自然か

**完了の美学について：**
- 完了画面で「満足した・終わった」という感覚が得られるか
- 「今日はここまで」ボタンを実際に使いたくなるか
- 完了後の待機画面は静かさを感じるか、空虚さを感じるか

**インタラクションについて：**
- いいね（数字なし）は承認欲求への影響が少ないか。押す行為の意味が変わるか
- ブックマークは実際に使われるか。「後で読み返す」という行為として機能するか
- 返信・リポストがなくても不便に感じないか、すっきり感があるか

**環境との同期について：**
- 時間帯による色変化が「時刻を伝えられている」感を与えないか（それは避けたい）
- 朝・夜の質感の違いが使い方のリズムに影響するか

**全体として：**
- prototype-01・02 と比べて「もっと見たい」という衝動が減っているか
- 「読み終わった」体験が一日の中で繰り返し訪れることで、利用パターンが変わるか
- 操作の選択肢が少ないことが「すっきり感」か「物足りなさ」かどちらに感じるか
