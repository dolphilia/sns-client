## 目的

このプロトタイプで検証したいことは **2 つ**に絞る。

1. **LLM 吟味モデルの体験検証**  
   取得した投稿をそのまま表示するのではなく、ローカル LLM（qwen3.5:9b / Ollama）が 1 件ずつ吟味し、通過した投稿だけを順に届ける設計が、どの程度ストレスを軽減するかを体感する。「待つ」ことのストレスより「届く」ことの小さな喜びが上回るかどうかを確かめる。

2. **漸進的な号の体験検証**  
   prototype-03 の「スナップショットとして固定」という号モデルを発展させ、「届いたものが積み重なって号になる」という漸進的な組み立て体験が、どのような印象を与えるかを確かめる。

---

## 核心コンセプト：LLM 吟味モデル

### 従来（prototype-03）との違い

```
prototype-03:
「読み始める」→ N件を一括取得 → フィルタ適用 → 号として固定 → 読書開始

prototype-04:
「読み始める」→ 多めに取得 → LLMが1件ずつ吟味 → 通過した投稿が届く → 読み続けられる
                                                        ↑ここを楽しめるかが鍵
```

### 「届く」という体験

LLM の処理を「フィルタリング」ではなく「**選書**」として設計する。  
本屋や図書館の司書が棚から選んだ本が届くような感覚。却下された投稿は視界に入らない。

- フィルタリングされた件数は表示しない（何が除外されたかを意識させない）
- 通過した投稿の件数だけが静かに増える
- 1 件届くたびに、小さく静かな到着演出がある

## prototype-03 からの継承と変更

### 引き継ぐもの

| 要素 | 内容 |
|------|------|
| **号（Edition）の概念** | 明示的操作で始まり、完了で終わる読書の単位 |
| **「読み始める」** | 号を開始するための儀式的なボタン操作 |
| **完了の美学** | 自然完了・任意完了・完了画面のデザイン原則 |
| **環境との同期** | 時間帯に応じた背景色（Ambient Sync） |
| **インタラクション最小化** | いいね・ブックマークのみ。返信・リポスト・引用なし |
| **エンゲージメント数の非表示** | いいね数・リポスト数を見せない |
| **ブックマーク** | ローカル保存のみ |
| **設定** | 号サイズ・フィードソース・リポスト除外設定 |

### 変更・発展させるもの

| 要素 | prototype-03 | prototype-04 |
|------|-------------|-------------|
| **フィルタリング主体** | mDeBERTa（WASM）+ キーワード | ローカル LLM（Ollama）がメイン |
| **号の組み立て** | 取得後に一括フィルタ → 即座に固定 | LLMが吟味しながら漸進的に組み立て |
| **読書開始タイミング** | 号が完全に固定されてから | 最初の数件が届いた時点から読み始められる |
| **待機表現** | 「読み始める」→即表示 | 「読み始める」→吟味アニメーション→届き始め |
| **号の完成** | 操作と同時に確定 | 吟味が完了した時点で確定 |

### 整合性の取れない部分（引き継がない）

- **「完全な固定スナップショット」の約束**：漸進的に届く性質上、読書開始時点では号がまだ完成していない。ただし吟味完了後は固定される。
- **Layer 1（mDeBERTa）のフィルタ**：LLM に一本化するため廃止。キーワードフィルタ（Layer 0）は引き続き前段として使用。

## 状態設計

### 吟味フロー

```
アプリ起動
  ├─ 号あり（reading / screening 継続中）  → 吟味画面（続き）or 読書継続
  ├─ 号あり（completed）                  → 待機画面
  └─ 号なし                              → 待機画面

待機画面
  └─ 「読み始める」
       ↓
  [fetching] Bluesky API からまとめて取得（目標件数の 2〜3 倍）
       ↓
  [screening] LLM が 2 件並列で吟味開始
       ↓（1 件通過するたびに届く）
  [arriving]  通過した投稿が順次追加される（読み始めてよい）
       ↓（吟味完了）
  [reading]   号が確定・固定。通常の読書モードへ
       ↓
  自然完了 or 任意完了
       ↓
  完了画面 → 待機画面へ
```

### ScreeningStore の設計

```typescript
type ScreeningState =
  | "idle"       // 待機中
  | "fetching"   // API から投稿取得中
  | "screening"  // LLM が吟味中（投稿が届き始めている）
  | "reading"    // 吟味完了・号が固定・読書中
  | "completed"  // 読み終わり

type ScreeningStore = {
  state: ScreeningState

  // 吟味の進行状況
  fetchedPosts: PostView[]    // API から取得した全投稿
  approvedPosts: PostView[]   // LLM を通過した投稿（順次追加）
  screenedCount: number       // 吟味が完了した投稿の数（通過 + 却下）
  isScreeningDone: boolean    // 全件の吟味が完了したか

  // 読書位置
  readUntilIndex: number
  completionType: "natural" | "manual" | null

  // アクション
  startEdition: () => Promise<void>  // 「読み始める」
  markRead: (index: number) => void
  completeEdition: (type: "natural" | "manual") => void
  reset: () => void
}
```

永続化キー: `bsky-screening`

### 件数の設計

| 設定 | 目標承認件数 | 取得する件数 |
|------|------------|------------|
| 小（10件） | 10件 | 30件 |
| 中（20件） | 20件 | 50件 |
| 大（30件）※デフォルト | 30件 | 80件 |
| 特大（50件） | 50件 | 120件 |

LLM の通過率をおおよそ 50〜70% と仮定して余裕を持って取得する。  
吟味中に目標件数に達した場合は残りの吟味を打ち切り、号を確定する。

## UI 設計

### 1. 待機画面（Waiting Screen）

prototype-03 から引き継ぐ。変更点は「読み始める」ボタン押下後の遷移先のみ。

```
─────────────────────────────────

  今日の号はまだありません


         [読み始める]

─────────────────────────────────
```

### 2. 吟味画面（Screening Screen）

**設計の核心**：LLM が吟味中であることを、ストレスなく伝える画面。

#### 言語の設計

| 状態 | 表示テキスト | 意図 |
|------|------------|------|
| fetching | 「今日の投稿を集めています」 | 中立・静か |
| screening（0件通過） | 「吟味しています…」 | 能動的・信頼感 |
| screening（N件通過） | 「N件 届きました」 | 到着の喜び。増えることが肯定的 |
| screening 完了 | 「今日の号が揃いました」 | 完成の満足感 |

#### 言語の禁止事項

- ❌「フィルタリング中」「除外しました」← 何かを排除しているイメージ
- ❌「N件中M件処理済み」← 進捗の遅さが気になる
- ❌「しばらくお待ちください」← 待たされている感
- ❌「残り X 件」← カウントダウンが焦りを生む

#### アニメーション設計

```
──── 吟味画面のレイアウト ────

  [ブランク空間]

  ゆっくり呼吸するドット
  ● ● ●  ← 3つが交互に明滅（breathing pulse）

  今日の号を吟味しています

  ──── 投稿が届くたびにここから ────

  ┌──────────────────────────┐  ← fade-in で静かに現れる
  │ [投稿カード 1]            │
  └──────────────────────────┘
  ┌──────────────────────────┐  ← 少し遅れて次が現れる
  │ [投稿カード 2]            │
  └──────────────────────────┘

  まだ届いています…  ← 吟味完了まで表示（小さく・薄く）
```

**呼吸アニメーション（CSS）**:
```css
@keyframes breathe {
  0%, 100% { opacity: 0.3; transform: scale(0.9); }
  50%       { opacity: 1.0; transform: scale(1.1); }
}
.dot-1 { animation: breathe 2.4s ease-in-out infinite 0.0s; }
.dot-2 { animation: breathe 2.4s ease-in-out infinite 0.4s; }
.dot-3 { animation: breathe 2.4s ease-in-out infinite 0.8s; }
```

**投稿到着アニメーション（CSS）**:
```css
@keyframes arrive {
  from { opacity: 0; transform: translateY(12px); }
  to   { opacity: 1; transform: translateY(0); }
}
.post-arriving { animation: arrive 0.5s ease-out forwards; }
```

#### 読み始められるタイミング

- 最初の 3 件が届いた時点で、自然にスクロールして読み始めることができる
- 下にスクロールするほど、さらに届いた投稿が続いている
- 吟味中であることを示すインジケーター（「まだ届いています…」）は画面下部にのみ表示

### 3. 読書中（Reading Mode）

吟味が完了し号が固定された後の通常読書モード。prototype-03 の EditionReader を踏襲。

```
  [細いプログレスバー（数字なし）] ← 号の読み進み

  [PostCard 1]
  [PostCard 2]
  ...
  [PostCard N]

  ──── 号の終わり ────
  [今日はここまで] ← 任意完了ボタン
```

吟味完了と同時に画面最下部の「まだ届いています…」が消え、静かに読書モードに移行する。

### 4. 完了画面

prototype-03 から変更なし。

```
  読み終えました。              ← 自然完了
  今日はここまで。              ← 任意完了

  [閉じる]  [新しいフィードを取得する]
```

## フィルタリングパイプラインの変更

### prototype-03 のパイプライン

```
号取得 → キーワードフィルタ → mDeBERTa ML → 号を固定 → 表示
```

### prototype-04 のパイプライン

```
「読み始める」
    ↓
Bluesky API（多めに取得）
    ↓
キーワードフィルタ（Layer 0）    … 即時・確実な除外のみ
    ↓
Ollama LLM（Layer 1・メイン）   … 2件並列・think:false
    ├─ 通過 → approvedPosts に追加 → 画面に届く
    └─ 却下 → silentlyDropped（表示なし、カウントも非表示）
    ↓
目標件数に達する or 取得済み全件を吟味完了
    ↓
号を固定 → 読書モードへ
```

### LLM 設定の引き継ぎ

- モデル: `qwen3.5:9b`（変更なし）
- `think: false`（変更なし）
- `temperature: 0.1`（変更なし）
- 並列数: 2 件同時（S-1 改善案の適用）
- Few-shot プロンプト: 追加（A-1 改善案の適用）

## 完成の定義

- 「読み始める」→ 吟味アニメーション → 投稿が届き始める → 読書 → 完了 の一連のフローが動作する
- LLM を通過した投稿のみが表示され、却下された投稿・件数は見えない
- 吟味中のアニメーションがストレスではなく「静かな期待感」を与える（体感できる）
- 吟味完了後に号が固定され、それ以降は自動更新されない
- 完了画面が静かに表示され、「終わった」感覚が得られる

## 次のプロトタイプに向けた判断軸

**吟味モデルについて：**
- 「届く」という体験は「待たされている」より「選んでもらっている」に感じるか
- 吟味中に読み始められることは良いか、それとも「全部揃ってから読みたい」か
- 吟味時間（1件あたり 3〜5 秒）は許容範囲か、長すぎるか
- LLM によって除外された投稿を「見たかった」と思うか

**号の漸進的組み立てについて：**
- 届くたびに小さな満足感があるか、それとも落ち着かないか
- 号が「完成した」感覚は得られるか
- 吟味完了後の「固定」は意味のある区切りとして感じられるか

**全体として：**
- prototype-03 より「もっと見たい」という衝動がさらに減っているか
- 吟味という仕組みを「信頼できる」と感じるか、それとも「おせっかい」か
- 毎日の習慣として続けられる体験の重さか
